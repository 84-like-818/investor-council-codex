# WORKFLOW_CN.md

## 最快可复制的 6 步法

### 1. 定位这个人物“负责什么，不负责什么”
不要上来就写口吻。先写职责边界。

建议用这一句：
> 这个人物是我的什么助手？他最擅长什么？绝不负责什么？

例如：
- Livermore：趋势、时机、仓位纪律、人性诊断；不负责估值、法律、税务、基本面建模
- 某企业家：产品判断、组织设计、增长节奏；不负责法规、会计、HR制度细节

### 2. 把资料拆成 4 层
- 原始资料：书、访谈、演讲、股东信、历史报道
- 二手研究：传记、研究文章、评论
- 现代转译：把老术语翻成今天可执行语言
- 你的讨论沉淀：你和 Codex 后续反复打磨后的结论

### 3. 先写 references，不急着写 SKILL
顺序建议：
1. `01-role-and-boundary.md`
2. `02-voice-and-style.md`
3. `03-doctrine.md`
4. `04-anti-patterns.md`
5. `05-modern-translation.md`
6. `06-evals.md`

### 4. 再写 SKILL.md
SKILL 只做 3 件事：
- 告诉 Codex 什么时候该触发
- 告诉 Codex 先读哪些文件
- 告诉 Codex 输出格式

### 5. 每轮对话后，做一次 distill
每次你和人物聊完，不要只留在聊天框里。
把有价值的内容沉淀成：
- 新增原则
- 新增反模式
- 新增“像本人”的语言片段
- 新增现代转译
- 新增测试问题

### 6. 稳定后，封成 plugin
本地能用 skill。
对外分发，用 plugin。

## 最重要的结构原则

### 原则 1：一个人物 = 一组文件，不是一段 prompt
因为人物要长期维护。

### 原则 2：人格和工作要分开
- 人格：像不像这个人
- 工作：这个人如何帮你做判断

### 原则 3：永远保留“反模式”
很多人物做得像不像，关键就在于：
> 他不会说什么？不会怎么判断？

### 原则 4：把你和 Codex 的后续讨论当成增量语料
但必须经过筛选与归档，不能原样灌回去。


## 推荐的真正复用点
不要把“人物”和“工作”绑死。

更好的做法是：
- 人物母包：这个人是谁、怎么说、怎么判断
- 工作子包：这个人以什么格式帮你工作

这样你后面扩人时，只要新建人物母包；
扩工作时，只要新建工作子包。

### 例子
- `livermore`：人物母包
- `livermore-chat`：对话助手
- `livermore-market-memo`：市场备忘录
- `livermore-journal-review`：交易复盘
